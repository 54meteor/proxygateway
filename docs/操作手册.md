## 第1章：系统简介

### 1.1 项目背景与定位

AI Gateway 是一个大模型流量中继平台（LLM Gateway），部署在客户端与各大模型服务提供商之间，实现以下核心价值：

- **统一接入层**：为内部系统提供单一的 LLM 调用入口，屏蔽多模型差异
- **用量追踪**：精确记录每次 API 调用的 Token 消耗，支持按用户、按日期多维度统计
- **计费管理**：支持按 Token 用量自动扣费，配合用户余额体系实现精细化成本控制
- **API Key 管理**：支持多用户、多 Key 体系，Key 启用/禁用、速率限制等企业级管理能力

### 1.2 核心功能列表

| 功能模块 | 说明 |
|---------|------|
| LLM 代理转发 | 支持 MiniMax（已上线）、OpenAI 和 Anthropic（预留） |
| API Key 鉴权 | 基于 Bearer Token 的认证，Key 存储使用 SHA256 哈希 |
| Token 计费 | 实时计算 Prompt/Completion Token 费用并扣除余额 |
| 用量统计 | 记录每次请求的 Token 消耗，支持按用户、按日期查询 |
| 请求日志 | 完整记录每次请求/响应的 JSON 到独立日志文件 |
| 管理后台 | Web 界面管理用户、API Keys、查看用量统计 |
| 健康检查 | 检查数据库和上游服务状态 |

## 第2章：快速部署

### 2.1 环境要求

| 组件 | 版本要求 | 说明 |
|------|---------|------|
| Go | ≥ 1.18 | 推荐使用最新版 |
| SQLite | 内置驱动 | 无需单独安装，使用纯 Go 驱动 `glebarez/sqlite` |
| 操作系统 | Linux / Windows / macOS | 跨平台支持 |
| 磁盘空间 | ≥ 500MB | 视日志量而定 |

### 2.2 快速安装

#### 2.2.1 克隆代码

```bash
git clone <repository-url>
cd ai-gateway
```

#### 2.2.2 配置环境变量

**Linux / macOS / WSL：**

```bash
export MINIMAX_API_KEY="your-minimax-api-key-here"
# 可选：
export OPENAI_API_KEY="your-openai-api-key-here"
export ANTHROPIC_API_KEY="your-anthropic-api-key-here"
```

**Windows (PowerShell)：**

```powershell
$env:MINIMAX_API_KEY="your-minimax-api-key-here"
```

#### 2.2.3 编译运行

```bash
# Linux / WSL
go build -o server cmd/server/main.go
./server

# Windows
go build -o server.exe cmd\server\main.go
server.exe
```

服务默认监听 `0.0.0.0:8080`。

#### 2.2.4 管理后台前端（可选）

如果需要使用 Vue 版管理后台（推荐），需要构建并独立部署：

```bash
cd admin-web
pnpm install
pnpm build
# 构建产物在 dist/ 目录，可使用 nginx 部署
```

> **注意**：Go 服务本身也内嵌了一套基于 HTML 模板的管理后台（访问 `/admin-dashboard` 等路径），Vue 前端是功能更完整的替代版本。

### 2.3 Docker 部署

#### Go 服务（主服务）

项目根目录没有提供 Dockerfile，可参考以下方式构建：

```dockerfile

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

RUN CGO_ENABLED=0 GOOS=linux go build -o server cmd/server/main.go

FROM alpine:latest
```

## 第3章：配置说明

### 3.1 config.yaml 完整配置项解释

```yaml
# ============ 服务配置 ============
server:
  host: "0.0.0.0"        # 监听地址，0.0.0.0 表示所有网卡
  port: 8080             # 监听端口
  mode: "debug"          # 运行模式：debug / release

# ============ 数据库配置 ============
database:
  driver: "sqlite3"      # 驱动类型（目前仅支持 sqlite3）
  path: "./ai_gateway.db" # SQLite 数据库文件路径（相对于可执行文件）

# ============ 大模型配置 ============
models:
  # OpenAI 配置（预留）
  openai:
    enabled: false                    # 是否启用
    base_url: "https://api.openai.com/v1"  # API 基础地址
    api_key: "${OPENAI_API_KEY}"      # 从环境变量 OPENAI_API_KEY 读取
    timeout: 120                     # 请求超时时间（秒）

  # Anthropic 配置（预留）
  anthropic:
    enabled: false
    base_url: "https://api.anthropic.com"
    api_key: "${ANTHROPIC_API_KEY}"
    timeout: 120

  # MiniMax 配置（已上线）
  minimax:
    enabled: true
    base_url: "https://api.minimaxi.com/v1"  # MiniMax API 地址
    api_key: "${MINIMAX_API_KEY}"   # 从环境变量 MINIMAX_API_KEY 读取
    timeout: 120                    # 请求超时时间（秒）

# ============ 计费配置 ============
# 单位：$/1K tokens（每千 Token 美元价格）
pricing:
  "gpt-4":
    prompt: 0.03        # Prompt Token 单价
    completion: 0.06    # Completion Token 单价
  "gpt-4o":
    prompt: 0.0025
    completion: 0.01
  "gpt-3.5-turbo":
    prompt: 0.0015
    completion: 0.002
  "claude-3-opus":
    prompt: 0.015
    completion: 0.075
  "claude-3-sonnet":
    prompt: 0.003
    completion: 0.015
  "MiniMax-M2.5":        # MiniMax 模型配置
    prompt: 0.01
    completion: 0.01

# ============ 限流配置 ============
rate_limit:
  enabled: true                    # 是否启用限流（注意：当前为简化实现）
  requests_per_minute: 60          # 每分钟最大请求数
  tokens_per_minute: 100000         # 每分钟最大 Token 数
```

### 3.2 环境变量

| 变量名 | 必需 | 说明 |
|--------|------|------|
| `MINIMAX_API_KEY` | **必需**（若启用 MiniMax） | MiniMax API Key |
| `OPENAI_API_KEY` | 可选 | OpenAI API Key（仅在 `models.openai.enabled: true` 时需要） |
| `ANTHROPIC_API_KEY` | 可选 | Anthropic API Key（仅在 `models.anthropic.enabled: true` 时需要） |

环境变量优先级：**环境变量 > config.yaml 中的 `${VAR}` 占位符 > config.yaml 中的明文值**

### 5.3 SDK 使用示例

#### 5.3.1 Python（OpenAI SDK 兼容方式）

AI Gateway 的 `/v1/chat/completions` 接口兼容 OpenAI ChatML 格式，可使用 OpenAI Python SDK 调用：

```python
from openai import OpenAI

client = OpenAI(
    api_key="YOUR-API-KEY",          # AI Gateway 的 API Key
    base_url="http://localhost:8080/v1"  # AI Gateway 地址
)

response = client.chat.completions.create(
    model="MiniMax-M2.5",
    messages=[
        {"role": "user", "content": "你好"}
    ],
    max_tokens=100
)

print(response.choices[0].message.content)
print(f"Total tokens: {response.usage.total_tokens}")
```

#### 5.3.2 JavaScript / Node.js

```javascript
// 使用 OpenAI JS SDK
import OpenAI from 'openai';

const client = new OpenAI({
  apiKey: 'YOUR-API-KEY',
  baseURL: 'http://localhost:8080/v1',
});

async function main() {
  const response = await client.chat.completions.create({
    model: 'MiniMax-M2.5',
    messages: [
      { role: 'user', content: '你好' }
    ],
    max_tokens: 100,
  });

  console.log(response.choices[0].message.content);
  console.log(`Total tokens: ${response.usage.total_tokens}`);
}

main();
```

## 第6章：计费说明

### 6.1 Token 计算方式

计费基于 LLM 返回的 `usage` 字段中的 Token 数量：

| 字段 | 说明 |
|------|------|
| `prompt_tokens` | 输入（用户消息）的 Token 数 |
| `completion_tokens` | 输出（AI 回复）的 Token 数 |
| `total_tokens` | 总 Token 数 |

### 6.2 价格配置方法

在 `config.yaml` 的 `pricing` 节点下，按模型名称配置：

```yaml
pricing:
  "MiniMax-M2.5":
    prompt: 0.01      # $ / 1K prompt tokens
    completion: 0.01  # $ / 1K completion tokens
```

### 6.3 费用扣除流程

```
1. 用户发起 /v1/chat/completions 请求
2. Gateway 调用上游 LLM，获取响应（含 usage）
3. BillingService 根据模型单价计算费用：
   cost = (prompt_tokens / 1000) * prompt_price
        + (completion_tokens / 1000) * completion_price
4. 原子性扣除用户余额（先查余额，不够则失败）
5. 记录 token_usage 表
6. 返回响应给客户端
```

### 6.4 设置不同模型价格

在 `config.yaml` 中添加或修改 `pricing` 配置即可：

```yaml
pricing:
  "MiniMax-M2.5":
    prompt: 0.01
    completion: 0.01
  "gpt-4o":
    prompt: 0.0025
    completion: 0.01
```

> **注意**：计费服务使用配置中的价格，`billing.go` 中有硬编码的默认值 `0.01`，但最终以 `config.yaml` 中配置的 `pricing` 为准。

## 第7章：日志说明

### 7.1 数据库日志

#### 7.1.1 token_usage 表结构

```sql
CREATE TABLE token_usage (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id          TEXT    NOT NULL,
    api_key_id       TEXT    NOT NULL,
    model            TEXT    NOT NULL,
    prompt_tokens    INTEGER NOT NULL,
    completion_tokens INTEGER NOT NULL,
    cost             REAL    NOT NULL,
    created_at       TEXT    DEFAULT (datetime('now'))
);

-- 索引
CREATE INDEX idx_token_usage_user_id ON token_usage(user_id);
CREATE INDEX idx_token_usage_created_at ON token_usage(created_at);
```

#### 7.1.2 查询用量的 SQL 示例

```sql
-- 查询指定用户的所有用量记录
SELECT * FROM token_usage WHERE user_id = 'user-uuid' ORDER BY created_at DESC;

-- 按日期统计每日用量
SELECT
    date(created_at) as date,
    model,
    COUNT(*) as requests,
    SUM(prompt_tokens) as total_prompt,
    SUM(completion_tokens) as total_completion,
    SUM(cost) as total_cost
FROM token_usage
WHERE date(created_at) BETWEEN '2026-03-01' AND '2026-03-31'
GROUP BY date(created_at), model
ORDER BY date DESC;

-- 统计总费用
SELECT
    model,
    COUNT(*) as requests,
    SUM(prompt_tokens) as total_prompt,
    SUM(completion_tokens) as total_completion,
    SUM(cost) as total_cost
FROM token_usage;
```

### 7.2 文件日志

#### 7.2.1 日志目录结构

```
ai-gateway/
├── gateway.log              # 应用运行日志（Info/Warn/Error）
├── logs/
│   ├── chat_2026-04-08.log  # 每日聊天详细日志
│   ├── chat_2026-04-09.log
│   └── ...
```

#### 7.2.2 gateway.log 格式

```
[2026-04-08 13:14:00] [INFO]  Database initialized: ./ai_gateway.db
[2026-04-08 13:14:01] [INFO]  Starting server on 0.0.0.0:8080
[2026-04-08 13:15:10] [INFO]  Request: userID=xxx, model=MiniMax-M2.5, messages=1
[2026-04-08 13:15:11] [INFO]  Response: userID=xxx, model=MiniMax-M2.5, prompt=42, completion=128, total=170, cost=0.001700
```

#### 7.2.3 chat_YYYY-MM-DD.log 格式（JSON Lines）

每次请求独立一条 JSON 记录：

```json
{
  "time": "2026-04-08 13:15:10",
  "user_id": "550e8400-...",
  "api_key_id": "7a1b2c3d-...",
  "model": "MiniMax-M2.5",
  "prompt_tokens": 42,
  "completion_tokens": 128,
  "total_tokens": 170,
  "cost": 0.001700,
  "request": { ... },
  "response": { ... }
}
```

#### 7.2.4 按日期分割说明

- `gateway.log`：不自动分割，需要使用 `logrotate` 等工具
- `chat_YYYY-MM-DD.log`：**自动按日期分割**，每天生成一个新文件

#### 8.1.1 服务启动失败

**错误：`failed to load config`**

```
# 原因：config.yaml 不存在或格式错误

1. 确认 config.yaml 在运行目录下
2. 检查 YAML 格式是否正确（注意缩进、空格）
3. 查看 gateway.log 获取详细错误
```

**错误：`failed to connect database`**

```
# 原因：SQLite 数据库文件无法创建或访问
# 解决：
1. 确认运行目录有写权限
2. 检查 config.yaml 中 database.path 路径
3. 尝试手动创建目录：mkdir -p data
```

**错误：`MiniMax API Key not configured`**

```
# 原因：未设置 MINIMAX_API_KEY 环境变量
# 解决：
export MINIMAX_API_KEY="your-key-here"
./server
```

#### 8.1.2 API 调用返回 401 Unauthorized

```
# 原因：API Key 无效或已禁用
# 排查步骤：
1. 检查 Authorization Header 是否正确：
   Authorization: Bearer YOUR-API-KEY
2. 确认 Key 未被禁用：
   curl http://localhost:8080/debug/keys
3. 确认用户余额充足（余额为负时仍可请求，但建议保持正余额）
```

#### 8.1.3 Token 计费异常

```
# 原因：配置了未知模型名称，导致无法匹配定价
# 排查：
1. 检查 config.yaml 中 pricing 配置的模型名称
2. 确认请求中 model 字段与 pricing 配置完全一致（区分大小写）
3. 查看 gateway.log 中是否有 "unknown model" 错误
```

#### 8.1.4 数据库损坏处理

```
# 错误：database is locked 或 SQLITE_CORRUPT
# 解决：
1. 停止服务
2. 备份损坏的文件：cp ai_gateway.db ai_gateway.db.bak
3. 尝试修复：sqlite3 ai_gateway.db ".recover" > ai_gateway_recovered.db
4. 替换文件：mv ai_gateway_recovered.db ai_gateway.db
5. 重启服务
```

### 8.2 备份与恢复

#### 8.2.1 数据库备份

```bash
# 冷备份（停止服务后执行）
cp ./ai_gateway.db ./backup/ai_gateway_$(date +%Y%m%d).db

# 热备份（无需停服）
sqlite3 ./ai_gateway.db ".backup './backup/ai_gateway_$(date +%Y%m%d).db'"

# 自动化备份（crontab 每日凌晨2点执行）
0 2 * * * cd /path/to/ai-gateway && sqlite3 ./ai_gateway.db ".backup './backup/ai_gateway_$(date +\%Y\%m\%d).db'"
```

#### 8.2.2 配置备份

```bash
# 备份配置文件（不包含明文 Key）
cp config.yaml config.yaml.bak

# 备份环境变量（建议使用密钥管理服务）
# 生产环境不要将真实 API Key 写在 config.yaml 中，使用环境变量
```

#### 8.2.3 恢复

```bash
# 停止服务
# 恢复数据库
cp ./backup/ai_gateway_20260401.db ./ai_gateway.db
# 恢复配置
cp config.yaml.bak config.yaml
# 重启服务
./server
```

### 8.3 性能调优

#### 8.3.1 SQLite 优化建议

| 优化项 | 方法 |
|--------|------|
| **定期 VACUUM** | `sqlite3 ai_gateway.db "VACUUM;"`（每月执行一次） |
| **Write-Ahead Log** | SQLite 默认使用 rollback journal，高并发时可考虑开启 WAL 模式 |
| **连接池** | 当前为单连接，高并发场景可考虑升级为 PostgreSQL |

```sql
-- 开启 WAL 模式（需停机执行）
PRAGMA journal_mode=WAL;
```

#### 8.3.2 并发限制说明

- Gin 默认并发处理，无需额外配置
- SQLite 写入为串行（单写锁），高频写入场景建议升级为 PostgreSQL
- 限流中间件（`rate_limit`）当前为简化实现（未实际限流），生产环境可接入 Redis 实现分布式限流
